# tg-summary-bot

Telegram-бот для автоматического создания конспектов статей. Отправьте ссылку боту в личные сообщения — он извлечет содержимое статьи, сгенерирует подробный технический конспект с помощью LLM и опубликует его в Telegram-канал с хештегами.

## Возможности

- Извлечение текста статей из любых URL (trafilatura)
- Генерация подробных конспектов через LLM (OpenRouter или пользовательский провайдер)
- Автоматические хештеги с накопительным пулом тегов
- Обнаружение дубликатов по нормализованным URL
- Поиск по хештегам и ключевым словам
- Очередь обработки с механизмом повторных попыток
- **Агрегатор контента**: автоматический сбор данных из внешних источников (RSS, имиджборды, API), генерация дайджестов и публикация в канал
- Расширяемая плагин-система для подключения новых источников данных
- Публикация конспектов и дайджестов в Telegram-канал
- Развертывание через Docker

## Требования

- Python 3.14+
- Docker и Docker Compose (для развертывания)
- Telegram Bot Token (от [@BotFather](https://t.me/BotFather))
- API-ключ OpenRouter ([openrouter.ai](https://openrouter.ai))

## Быстрый старт

### Локальная разработка

1. Клонируйте репозиторий:

```bash
git clone https://github.com/your-username/tg-summary-bot.git
cd tg-summary-bot
```

2. Создайте виртуальное окружение и установите зависимости:

```bash
python -m venv .venv
source .venv/bin/activate  # Linux/macOS
# .venv\Scripts\activate   # Windows
pip install -r requirements.txt
pip install -r requirements-dev.txt  # для разработки
```

3. Скопируйте `.env.example` в `.env` и заполните переменные:

```bash
cp .env.example .env
```

4. Запустите бота:

```bash
python -m src.main
```

### Docker

```bash
cp .env.example .env
# Заполните .env

docker compose up -d
```

## Переменные окружения

| Переменная | Обязательная | Описание |
|---|---|---|
| `TELEGRAM_BOT_TOKEN` | Да | Токен бота от @BotFather |
| `TELEGRAM_CHANNEL_ID` | Да | ID канала для публикации конспектов |
| `TELEGRAM_ADMIN_ID` | Да | Ваш Telegram user ID |
| `OPENROUTER_API_KEY` | Да | API-ключ OpenRouter |
| `CUSTOM_LLM_API_URL` | Нет | URL пользовательского LLM-провайдера |
| `CUSTOM_LLM_API_KEY` | Нет | API-ключ пользовательского провайдера |
| `CUSTOM_LLM_MODEL` | Нет | Название модели пользовательского провайдера |
| `LLM_REQUEST_DELAY` | Нет | Задержка между запросами к LLM в секундах (по умолчанию: 2) |
| `LLM_MAX_RETRIES` | Нет | Максимальное количество повторных попыток (по умолчанию: 3) |
| `DB_PATH` | Нет | Путь к файлу SQLite базы данных (по умолчанию: `data/bot.db`) |
| `AGGREGATOR_ENABLED` | Нет | Включить агрегатор (`true`/`false`, по умолчанию: `false`) |
| `AGGREGATOR_ENABLED_SOURCES` | Нет | Включённые источники через запятую (например: `rss_example`) |
| `AGGREGATOR_SCHEDULE` | Нет | Интервал в минутах или cron (`60` или `cron:0 9 * * *`) |
| `AGGREGATOR_TONE` | Нет | Пресет тона для дайджестов (по умолчанию: `default`) |

### Стратегия выбора LLM-провайдера

Если заданы все три переменные `CUSTOM_LLM_*`, используется пользовательский провайдер. В противном случае запросы идут через OpenRouter. Оба провайдера используют OpenAI-совместимый API.

## Команды бота

Бот работает только в личных сообщениях с администратором (`TELEGRAM_ADMIN_ID`). Фильтр применяется на уровне роутера.

| Команда | Описание |
|---|---|
| `/start` | Приветственное сообщение |
| `/search #тег` | Поиск статей по хештегу |
| `/search ключевое_слово` | Полнотекстовый поиск по конспектам |
| `/stats` | Статистика: количество статей, тегов, успешных/неудачных обработок |
| `/retry` | Повторить обработку всех неудачных статей |
| `/retry <id>` | Повторить обработку конкретной статьи |

## Как это работает

Обработчики бота только сохраняют статьи в базу данных. Вся обработка выполняется фоновым процессором очереди.

**При получении ссылки:**
1. Пользователь отправляет ссылку боту в ЛС (доступно только администратору)
2. Бот извлекает URL из сообщения
3. Проверяет наличие дубликатов в базе данных
4. Сохраняет запись со статусом `pending` и `chat_id` пользователя
5. Отвечает пользователю: "Статья добавлена в очередь"

**Фоновая обработка (каждые 10 секунд):**
6. Процессор берет статьи из очереди и обрабатывает последовательно
7. Загружает HTML страницы и извлекает текст через trafilatura
8. Отправляет текст + топ-50 хештегов в LLM для генерации конспекта
9. Новые хештеги добавляются в пул
10. Конспект публикуется в канал
11. Пользователь получает подтверждение (через сохраненный `chat_id`)

## Формат публикации в канале

```
📎 Заголовок статьи

Подробный технический конспект — не тизер, а полноценный
дайджест ключевой информации, чтобы читателю не нужно было
открывать оригинал.

#тег1 #тег2 #тег3

🔗 Ссылка на оригинал
```

Язык конспектов — всегда русский (статьи на других языках переводятся автоматически).

## Развертывание

### Docker Compose (рекомендуется)

```bash
docker compose up -d
```

SQLite база данных сохраняется через Docker volume.

### GitHub Actions CI/CD

Проект настроен для автоматического развертывания:

1. Push в `main` запускает GitHub Actions workflow
2. Собирается Docker-образ и публикуется в GHCR
3. По SSH подключается к VPS, выполняет pull и перезапуск контейнера

Переменные для CI/CD настраиваются в GitHub Secrets.
